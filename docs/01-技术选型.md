# 技术选型（Coze → 本地开源）

本节用于说明：原先在 Coze 上实现的“合规智能体 + 人机协同智能体”如何迁移为可开源、可本地运行、可复现的实现。

## 选型结论（推荐）

- **核心实现：Python + LangGraph（LangChain 生态）**
  - 原因：代码形态清晰、可测试、可版本化；最容易与本地“感知质量智能体”以函数/Tool 的形式直接连接。
  - 多智能体编排建议采用“图/状态机”而不是纯 prompt 串联，便于表达：合规优先、不确定→复核、失败隔离等论文里的工程逻辑。
- **可选集成：Dify 作为非核心依赖**
  - 适合：你想要一个可视化工作流界面给演示/迭代 prompt。
  - 但不建议作为本仓库的唯一实现：会引入部署与平台耦合成本，且测试/复现更偏“平台工程”而不是“论文工程实现”。

> 简单理解：**仓库里要有“可复现的代码实现”（主线）**；Dify 可以作为后续附加能力（支线），而不是决定性依赖。

## Dify vs LangChain/LangGraph（面向本项目的取舍）

### 选择 LangGraph/LangChain 的场景

- 你希望本项目“开箱即跑”的本地 Demo 能完全由代码定义（可单元测试、可 CI）
- 需要将质量智能体作为工具直接调用（同进程/同虚拟环境），并与合规智能体共享统一 schema 与证据链
- 需要清晰表达路由逻辑（合规优先、不确定→复核、冲突仲裁、重试/降级）

### 选择 Dify 的场景

- 你主要诉求是“快速搭工作流 + 有 UI + 便于展示 prompt/知识库配置”
- 你愿意接受：需要 Docker 部署、平台版本差异、工作流以平台配置为主、测试更偏集成测试

## 与感知质量智能体的连接方式（推荐接口）

- 将质量智能体封装为一个纯函数/类方法：输入 `ModerationItem`，输出 `QualitySignal`（含相似证据与分值）
- 在 LangGraph 中把质量智能体注册成 Tool/Node，与合规智能体并行或串行调用
- 结果整合节点只依赖 schema，不依赖具体实现（便于将来替换合规模块：规则/mock/LLM/RAG）

## 后端形态建议（最小但够用）

- **默认：命令行 Demo + 本地 JSONL 输入输出**（最利于复现与审查）
- **可选：FastAPI 提供 REST 接口**，便于未来接任何技术栈（包括 Java/旧系统）

> 本项目不建议第一阶段投入完整 B/S 系统（前端、权限、队列、数据库全套）。把“审核引擎 + 结构化结果 + 证据链”打磨好，价值更集中。
